# Textbook Authoring Guidelines

The textbook is generated by `scripts/build_textbook.py`. For every topic,
the generator first looks for an authored source file:

```
content/part-XX/<topic>/textbook.md
```

If it exists, that file becomes the topic's section in the book. If it does
not, the generator falls back to narration-derived prose with
auto-generated "practice checkpoints" — a stopgap, not a goal. Every topic
should eventually have an authored `textbook.md`.

A part-level opener can be authored in `content/part-XX/textbook-intro.md`;
it replaces the outline summary at the start of the chapter. For Part 8
(which has no slide topics yet) the intro file carries the whole chapter.

## What the file is

One topic's section of a real book: continuous, engaging prose that
teaches the topic properly. The slides are a skeleton, not a boundary —
explain everything a curious student would want explained, including the
things a lecturer would say out loud but never put on a slide.

- **Audience**: university students heading into IT jobs, plus early-career
  practitioners. Assume intelligence, not experience.
- **Length**: roughly 900–1,800 words per topic. Enough to actually teach;
  short enough to stay honest.
- **Voice**: direct, practical, occasionally wry. Australian context where
  it matters (employment, industry examples), international otherwise.
  Prefer "you" over "the student". Contractions are fine.
- **Ground it**: concrete workplace scenes, named (fictional) companies and
  people, realistic numbers, real tool names where the slides use them.
  The course already uses recurring characters (e.g. Sarah's startup in
  Part 6) — reuse them rather than inventing parallel casts.
- **Job-role thread**: the course's running theme is career readiness. Where
  natural, connect the topic to actual roles, seniority levels and how
  people move between them. Don't force it into every topic.
- **Cover the slides**: every substantive point on the topic's slides and in
  its narratives should appear somewhere in the prose, usually deepened.
  Never contradict the slides. Do not copy narrative dialogue verbatim —
  it is spoken two-speaker material; rewrite as prose.

## Markdown subset (hard contract)

The generator converts a deliberately small subset. Use only:

- Paragraphs separated by blank lines.
- `##` and `###` headings (rendered as subsection/subsubsection). Do NOT
  start the file with a title heading — the generator emits the section
  title from the slide deck. Two to five `##` headings per topic is typical.
- `- ` bullet lists and `1. ` numbered lists, one nesting level (indent two
  spaces). No blank lines between items of the same list.
- `**bold**`, `*italic*`, `` `inline code` ``.
- Fenced code blocks (``` ... ```) for genuinely code-like artifacts.
- `> ` block quotes for short asides, definitions or war stories.
- Images on their own line: `![Caption](images/file.png)` — only for files
  that actually exist in the topic's `images/` directory.

No tables, no HTML, no links (the book is print-first), no footnotes, no
`#` top-level headings. Anything else will be silently mangled.

## Slop patterns (rejected on sight)

- Motivational filler: "In today's fast-paced world…", "…is more important
  than ever", "journey", "delve", "landscape", "crucial" as a reflex.
- Bullet lists that restate the slide instead of teaching. If a list isn't
  earning its keep, write a paragraph.
- The same transition twice ("Let's explore…", "Now that we've covered…").
- Summary sections that merely repeat the section ("In conclusion…").
- Hedge-everything prose. Take positions: this course teaches judgement.
- For Part 7 (Indigenous data sovereignty and open source): follow the
  narratives' framing precisely, stay factual and respectful, and do not
  invent cultural details that are not in the source material.

## Structure that works

Open with something real — an incident, a decision, a cost, a person in a
role — then use it. Develop the ideas in prose with headings where the
topic genuinely shifts. Lists for genuinely enumerable things (checklists,
tiers, metric definitions). Close with the practical takeaway: what the
reader should now be able to do, notice, or argue for. A short `>` aside
with a war story or rule of thumb is welcome once or twice per topic.
